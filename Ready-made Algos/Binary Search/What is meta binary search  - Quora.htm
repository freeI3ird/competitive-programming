Meta binary search (also called one-sided binary search by Steven Skiena in The Algorithm Design Manual on page 134) is a modified form of binary search that incrementally constructs the index of the target value in the array. Like normal binary search, meta binary search takes O(lgn)O(lg⁡n) time.

The exact implementation varies, but the basic algorithm has two parts:

  1. Figure out how many bits are necessary to store the largest array index.
  2. Incrementally construct the index of the target value in the array by determining whether each bit in the index should be set to 1 or 0.

This topic is surprisingly difficult to describe, but the actual concept is very simple. I think the best explanation will be to go through an example.

Let's suppose we have the following array [-10, -5, 4, 6, 8, 10, 11] and we're searching for the value 10.

The first thing we need to do is figure out how many bits are necessary to store the largest array index. The length of the array is 7, so the largest array index is 6 (zero-based indexing).

The value 6 in binary is 110, so it requires 3 bits to represent it. We can easily calculate the value 3 by computing ⌈lg(n−1)⌉⌈lg⁡(n−1)⌉ = ⌈lg6⌉⌈lg⁡6⌉ = ⌈2.58⌉⌈2.58⌉ = 3.

The next step is to incrementally construct the index of the target value in the array. Looking at our array, the target value, 10, is at index 5.  Our goal is to construct the index 5, one bit at a time.

The way we construct the index 5 incrementally will involve a lot of thinking in binary. The value 5 in binary is 101. We will construct the value 101 incrementally, starting from the most significant bit (the leftmost one) and working towards the least significant bit (the rightmost one).

We know that we need three bits to represent an array index. Let's start with placeholders: _ _ _.

Now, let's iterate from most significant bit to least significant bit (left to right).

We first try to determine whether we need:

1 _ _

or:

0 _ _

In order for the most significant bit to be 1, the array index of the target value must be greater than or equal to 100 in binary.  The value 100 in binary is the decimal value 4.  So we look at array[4] to determine whether that index overshoots or undershoots the array value we're looking for.  The value of array[4] is 8, so we've underestimated the index of the target value (larger indices produce larger values, and we're looking for 10, which is larger). Since we know that the target index must be greater than 4, the most significant bit must be 1 (or else we won't be able to construct a target index larger than 4).

We repeat this process using the intermediary result: 1 _ _.

The value of array[6] is 11, so we've overestimated the index of the target value (we're looking for 10, which is smaller). Since we know that the target index must be less than 6, the second most significant bit must be 0 (or else we won't be able to construct a target index smaller than 6).

Meta binary search (also called one-sided binary search by Steven Skiena in The Algorithm Design Manual on page 134) is a modified form of binary search that incrementally constructs the index of the target value in the array. Like normal binary search, meta binary search takes O(lgn)O(lg⁡n) time.

The exact implementation varies, but the basic algorithm has two parts:

  1. Figure out how many bits are necessary to store the largest array index.
  2. Incrementally construct the index of the target value in the array by determining whether each bit in the index should be set to 1 or 0.

This topic is surprisingly difficult to describe, but the actual concept is very simple. I think the best explanation will be to go through an example.

Let's suppose we have the following array [-10, -5, 4, 6, 8, 10, 11] and we're searching for the value 10.

The way we construct the index 5 incrementally will involve a lot of thinking in binary. The value 5 in binary is 101. We will construct the value 101 incrementally, starting from the most significant bit (the leftmost one) and working towards the least significant bit (the rightmost one).

We know that we need three bits to represent an array index. Let's start with placeholders: _ _ _.

Now, let's iterate from most significant bit to least significant bit (left to right).

We first try to determine whether we need:

1 _ _

or:

0 _ _

In order for the most significant bit to be 1, the array index of the target value must be greater than or equal to 100 in binary.  The value 100 in binary is the decimal value 4.  So we look at array[4] to determine whether that index overshoots or undershoots the array value we're looking for.  The value of array[4] is 8, so we've underestimated the index of the target value (larger indices produce larger values, and we're looking for 10, which is larger). Since we know that the target index must be greater than 4, the most significant bit must be 1 (or else we won't be able to construct a target index larger than 4).

We repeat this process using the intermediary result: 1 _ _.

Now we try to determine whether we need:

1 1 _

or:

1 0 _

In order for the second most significant bit to be 1, the array index of the target value must be greater than or equal to 110 in binary. The value 110 in binary is the decimal value 6. So we look at array[6] to determine whether that index overshoots or undershoots the array value we're looking for. The value of array[6] is 11, so we've overestimated the index of the target value (we're looking for 10, which is smaller). Since we know that the target index must be less than 6, the second most significant bit must be 0 (or else we won't be able to construct a target index smaller than 6).

We repeat this process using the intermediary result: 1 0 _.

Now we try to determine whether we need:

1 0 1

or:

1 0 0

In order for the third most significant bit (the least significant bit) to be 1, the array index of the target value must be greater than or equal to 101 in binary. The value 101 in binary is the decimal value 5. So we look at array[5] to determine whether that index overshoots or undershoots the array value we're looking for. The value of array[5] is 10, so we've found the value we're looking for! We can return index 5 as the answer.

Now, there are two conditions I haven't mentioned.

One condition that can happen is that the target index we generate can be larger than length of the array. We must be mindful of this possibility. If the target index we generate is beyond the length of the array, we know that we want to choose 0 for the current bit position instead of 1.

The other condition is that the array value we're looking for isn't in the array. In this case, the target value will never be found. If we exit the loop without finding the value we're looking for, then we can return a sentinel indicating that the target value was not found.

And that's the algorithm!

So the next question you might ask is: "Why would anyone use meta binary search?"

The advantage of meta binary search over normal binary search is that it's easier to code correctly without making off-by-one errors or overflow errors. Because of this, many competitive programmers use it instead of normal binary search.

Meta binary search also has another advantage: it can be easily modified to work on arrays of unknown length. You just modify both parts of the code to handle array out of bounds exceptions instead of relying on the element count.

The disadvantage of meta binary search over normal binary search is that there are two parts that take O(lgn)O(lg⁡n) time.  Therefore, the algorithm runs at roughly half the speed of normal binary search.

Here's some Java code I wrote up for it:

  1. public class MetaBinarySearch {
  2.     public static void main(String[] args) {
  3.         int index = metaBinarySearch(new Integer[] { -10, -5, 4, 6, 8, 10, 11 }, 10);
  4.  
  5.         System.out.println(index);
  6.     }
  7.  
  8.     public static <T extends Comparable<T>> int metaBinarySearch(T[] values, T valueToFind) {
  9.         if (values == null) {
  10.             throw new IllegalArgumentException("Cannot binary search null array!");
  11.         }
  12.  
  13.         if (valueToFind == null) {
  14.             throw new IllegalArgumentException("Cannot binary search for null element!");
  15.         }
  16.  
  17.         int n = values.length;
  18.         int numBitsNeededForMaxIndex = 0;
  19.  
  20.         while ((1 << numBitsNeededForMaxIndex) < n) {
  21.             ++numBitsNeededForMaxIndex;
  22.         }
  23.  
  24.         int index = 0;
  25.  
  26.         for (int shift = numBitsNeededForMaxIndex - 1; shift >= 0; --shift) {
  27.             int newIndexCandidate = index + (1 << shift);
  28.  
  29.             if (newIndexCandidate >= n) {
  30.                 continue;
  31.             }
  32.  
  33.             int comparison = values[newIndexCandidate].compareTo(valueToFind);
  34.  
  35.             if (comparison == 0) {
  36.                 return newIndexCandidate;
  37.             }
  38.  
  39.             if (comparison < 0) {
  40.                 index = newIndexCandidate;
  41.             }
  42.         }
  43.  
  44.         return -1;
  45.     }
  46. }
